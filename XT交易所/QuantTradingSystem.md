## 量化交易系统｜框架、数据、执行与回测（XT）

### 技术栈与边界（简述）

- **语言与运行时**：Python（以异步 IO 为主），关键热点路径按性能需求做数据结构优化与内存复用。
- **数据与通信**：WebSocket（行情/回报订阅）、REST（快照/补偿/查询）、内部事件总线（策略与执行链路）。
- **存储与缓存**：MySQL/Redis（按业务阶段与性能约束选型），历史数据按时间分区管理。
- **工程目标**：低延迟、可恢复、可回放、可观测；极端行情下优先保护撤单、风控与资金安全相关链路。

### 1. 业务背景

交易所量化系统的目标并不是“跑策略”，而是围绕交易所业务约束建立一套可持续运行的体系：**行情高吞吐接入、策略低延迟响应、订单执行可控、风控约束前置、可观测与可追溯**。系统需要覆盖日常稳定行情与极端行情（插针、流动性骤降、撮合拥塞、网络抖动）两种状态，并保证资金与风险边界不被突破。

### 2. 我的角色与职责

- 参与并负责量化交易系统的核心研发与长期演进，覆盖行情、策略调度、订单管理、风险检查与交易所网关适配等关键链路。
- 主导多项策略从研究、回测、仿真到实盘的完整落地，负责策略的可运行性、稳定性与收益曲线健康度（回撤、滑点、成交率等）。
- 处理交易所环境下的性能与稳定性问题：事件驱动、异步 IO、热点数据结构、故障隔离、重试与幂等、压测与线上回放。

---

### 3. 系统总体架构（落地形态）

量化系统从“数据 → 决策 → 执行 → 风控 → 观测”组织，核心模块如下：

#### 3.1 行情接入与标准化（Market Data Ingestion）

- **接入形式**：以 WebSocket 为主，REST 为补充（启动时快照、异常时回补、对账校验）。
- **关键治理点**：
  - **协议解码与标准化**：不同交易对/不同通道的消息做统一 schema（trade / book / kline / ticker），便于策略层无感使用。
  - **顺序与一致性**：维护每个交易对的序列号/时间戳边界，处理乱序、丢包、重复消息；在必要时进行快照 + 增量的重建。
  - **热数据缓存**：Top-of-book、盘口增量、最新成交价等热点数据驻留内存（并配合 Redis/共享内存做多进程共享），让策略读取做到常量级开销。
  - **断线恢复**：自动重连、订阅恢复、回补窗口控制，避免断线后数据空洞导致策略误判。
  - **订阅管理**：按策略/交易对维度维护订阅关系，避免重复订阅放大带宽与解析成本；对无订阅的交易对及时释放资源。
  - **数据质量监测**：对延迟、缺口、回补次数、重连次数做指标化，异常波动触发告警与降级策略（例如暂停某类策略或降低频率）。

#### 3.2 事件驱动总线与策略调度（EventBus / Scheduler）

- **核心思想**：以“事件”为单位驱动策略，而不是轮询；策略收到事件后快速计算信号并进入下单路径。
- **调度策略**：
  - 按交易对/策略维度做隔离，避免某个策略阻塞全局；
  - 为策略设置超时、降级与熔断（极端行情下，优先保证风控与撤单可用）；
  - 支持策略的启停、参数热更新、灰度发布与回滚。
  - 支持“只读策略/交易策略”分级调度，避免数据类任务抢占交易线程与事件循环资源。

#### 3.3 订单管理系统（OMS）

- **职责**：对接交易所下单/撤单接口，维护订单生命周期（Created → Submitted → PartFilled/Filled/Cancelled/Rejected）。
- **关键实现**：
  - **幂等与去重**：重试场景下用 clientOrderId / idempotency key 保证不会重复下单。
  - **状态机**：订单状态变更有严格约束，异常状态（例如“已成交但回报丢失”）通过对账补齐。
  - **撤单优先级**：极端行情下撤单是保命动作，撤单通道与下单通道做优先级和限流隔离。
  - **回报处理**：撮合回报、成交回报、账户变化回报统一进入事件流，策略侧通过订阅获取实时状态。
  - **超时与补偿**：对“请求已发出但无回报”的订单建立超时扫描与补偿查询，避免订单长期悬挂影响策略判断。
  - **速率限制适配**：对不同接口的限频策略做统一封装（令牌桶/滑动窗口等），并支持按账户/交易对维度配置，降低触发平台限频的概率。

#### 3.4 风险检查前置（Pre-Trade Risk）

下单前必须经过风险检查，防止策略 bug 或行情异常导致资金风险扩大。常见检查项：

- **价格偏离**：与最新盘口/指数价的偏离超过阈值则拒单或改价；
- **数量/名义价值限制**：单笔/单日限额，防止异常放量；
- **仓位与杠杆约束**：开仓前检查保证金与风险限额；
- **自成交/循环下单**：对同一交易对的对敲风险做限制；
- **频率控制**：单位时间下单/撤单次数限制，防止触发交易所限频或被动风控。
 - **参数校验与白名单**：对策略参数（价格精度、数量精度、最小下单量）做统一校验与标准化，减少“参数不合法导致大量拒单”的故障形态。

#### 3.5 回测与仿真（Backtest / Simulation）

- **历史数据管理**：支持 tick 级/分钟级数据回放；建立统一的数据格式，确保回测与实盘一致的输入语义。
- **撮合仿真**：提供可配置的撮合模型（滑点、手续费、成交概率、深度影响），让回测结果更贴近真实。
- **绩效指标**：夏普、最大回撤、胜率、换手、滑点成本、成交率等；并对策略的稳定性做压力测试（极端行情回放）。
 - **一致性校验**：同一策略在回测/仿真/实盘三种模式下保持统一接口与数据语义，减少“回测可行、实盘不可用”的落差。

### 4. 代表性策略落地

#### 4.1 三角套利（Triangle Arbitrage）

- **核心逻辑**：实时监控多个交易对之间的价格偏差，计算可执行的套利路径，在满足手续费与滑点约束下进行对冲成交。
- **工程要点**：
  - 路径搜索与权重计算（把手续费、滑点、成交概率纳入权重）；
  - 多腿订单的执行编排：先后顺序、并发度控制、失败回滚与补仓；
  - 在极端行情下做快速止损与撤单，防止单腿成交导致裸露风险。

#### 4.2 镜像做市（Market Making）

- **核心目标**：在控制库存风险前提下提供持续报价，提高成交与手续费收益。
- **工程要点**：
  - 盘口快照同步：避免策略层频繁复制大对象，减少延迟与内存开销；
  - 报价刷新节奏：根据波动率与盘口变化动态调整刷新频率，降低无效撤单；
  - 风险护栏：库存上限、价差上限、异常波动熔断。

#### 4.3 跨市场套利（Cross-Market Arbitrage）

- **核心逻辑**：跟踪不同市场/不同合约的价差，按风险约束分批执行对冲。
- **工程要点**：
  - 交易所 API 限频适配：拆单、分批、退避重试；
  - 弱网/延迟场景下的保护：价差过期判定、成交回报超时处理；
  - 对冲失败兜底：补仓与撤单策略，避免暴露头寸。

### 5. 稳定性与可观测性

交易系统的“可用”不等于“能跑”，需要做到故障可控与可恢复：

- **链路监控**：行情延迟、事件堆积、下单耗时、撤单耗时、回报延迟、失败率等指标化；
- **日志与追踪**：以订单/策略/交易对为主键做可追溯日志，支持事后复盘；
- **故障隔离**：策略隔离、通道隔离（下单/撤单/回报）、限流与熔断；
- **压测与回放**：定期用历史极端行情进行回放压测，验证系统在峰值情况下的行为是否符合预期。

---

### 6. 交付效果与业务价值（可量化指标口径）

- **策略研发效率**：统一框架与调度后，新策略从接入、回测、仿真到上线的路径更短，减少重复造轮子与联调成本。
- **风险可控**：风控前置与撤单优先保证了极端行情下“先保命、再赚钱”的策略底线。
- **系统稳定**：通过链路隔离、限频适配与对账补偿，降低了线上异常对资金与收益曲线的影响范围。
