## 购物车系统｜Redis 缓存、合并与一致性（喜宝）

### 技术栈与边界（简述）

- **核心存储**：Redis（实时读写与热点数据）、MySQL（兜底与恢复）。
- **一致性策略**：缓存优先、异步落库、对账补偿；并发写通过锁/串行化控制。
- **工程目标**：高并发下低延迟、可降级、可恢复；对下单链路提供稳定输入。

### 1. 业务特点与设计目标

购物车是电商系统里最典型的高频模块之一，具备“写多读多、强交互、实时性要求高”的特征。设计目标主要围绕三点：

- **性能**：大促场景下依然保持低延迟与高吞吐；
- **一致性**：用户侧展示结果一致，不出现明显的“数量回滚/勾选状态丢失”；
- **可恢复**：缓存异常或局部故障时可降级与恢复，避免影响下单主链路。

---

### 2. 总体架构：Redis + MySQL（缓存主存储、数据库兜底）

- **Redis**：承载购物车的实时读写与展示快照；
- **MySQL**：作为持久化备份与恢复来源，用于极端情况下的数据回补与对账；
- **MQ（可选/阶段性）**：用于异步落库与削峰，避免高峰写入直接压垮数据库。

---

### 3. Redis 数据结构设计

为兼顾“单字段更新效率”和“整车渲染速度”，购物车采用组合存储：

- **Hash（核心字段）**：按用户维度存储商品维度的数量、选中状态等可频繁变更字段  
  - Key 示例：`cart:items:{userId}`  
  - Field 示例：`{skuId: {"qty": 2, "checked": 1, "update_at": 1710000000}}`

- **String（渲染快照/聚合视图）**：存放购物车页面所需的聚合结果（减少多次读取与拼装）  
  - Key 示例：`cart:snapshot:{userId}`
  - 内容：sku 列表、金额汇总、优惠提示等（根据业务裁剪）

- **过期策略**：为不同用户购物车 key 设置随机过期时间，降低缓存雪崩概率；同时对活跃用户做续期或预热。

---

### 4. 典型操作链路与一致性策略

#### 4.1 加购

- **流程**：
  1) 校验商品是否可售、是否下架、库存是否可用（可走商品缓存/库存服务）；  
  2) 写入 Redis Hash（数量累加、选中状态默认勾选）；  
  3) 更新/重建渲染快照（同步更新或异步重建）；  
  4) 异步落库（如采用 MQ）。

- **一致性重点**：
  - 购物车展示以 Redis 为准，避免 DB 与缓存双读造成抖动；
  - 对“并发加购同一 sku”引入用户维度的串行化控制（例如 userId 级锁），避免数量覆盖。

#### 4.2 改数量 / 勾选 / 删除

这是最容易产生并发与一致性问题的路径。

- **写策略**：
  - 优先写 Redis（保证用户即时可见）；
  - 采用“先删快照 / 标记快照过期 → 更新 Hash → 重建快照”的方式，避免旧快照被继续读取；
  - 落库采用异步方式，失败可重试，并通过对账任务补齐。

- **并发控制**：
  - 粗粒度：按 userId 加锁，确保同一用户的购物车操作串行；
  - 细粒度：按 userId+skuId 加锁，提升并发能力（视业务与实现复杂度取舍）。
  - 锁实现一般使用 Redis 原子操作（SET NX + 过期）或 Lua 脚本保证原子性，并为异常退出设置自动过期避免死锁。

#### 4.3 购物车展示

- **读策略**：
  - 优先读取 `cart:snapshot:{userId}`，命中则直接返回；
  - 未命中则读取 Hash 并组装快照（可能需要补充商品价格/活动信息），再写回快照；
  - 对价格变动、活动变动设置快照失效条件，避免展示过期价格。
  - 对热点用户的快照重建加入互斥与短期空值缓存，避免缓存击穿导致的瞬时 DB/QPS 抖动。

---

### 5. 未登录态购物车与登录合并

未登录用户通常使用 cookie/本地存储承载临时标识（UUID），登录后需要与账户购物车合并：

- **临时标识**：`cart:items:guest:{uuid}` + `cart:snapshot:guest:{uuid}`
- **合并策略**：
  - 同一 sku：数量相加或以账户为准（按产品策略确定），并处理选中状态；
  - 不同 sku：直接合并；
  - 合并完成后清理 guest cart，避免重复。

---

### 6. 降级与恢复

购物车允许一定程度的最终一致性，但要保证下单链路可用：

- **Redis 不可用**：降级到 MySQL 读取（性能下降但可用），并限制部分高频写操作；
- **MQ 不可用**：落库改为同步或写入本地队列，恢复后回放；
- **数据对账**：定时任务对比 Redis 与 MySQL 差异，补齐落库失败的变更（按用户/时间窗口批处理）。

---

### 7. 可观测与运维要点

- **关键指标**：加购/改数量/展示接口耗时、Redis 命中率、快照重建次数、加锁等待时间、落库延迟、失败重试次数。
- **告警与排障**：Redis 响应时间飙升、命中率骤降、快照重建暴涨、DB 慢查询增加等事件可快速定位到具体链路。

---

### 8. 结果与收益

- 大促期间购物车链路数据库压力显著下降，接口耗时更稳定；
- 用户侧交互更顺滑，改数量与勾选响应更快；
- 通过锁与快照策略减少并发覆盖与展示不一致问题；
- 购物车与下单链路的边界更清晰，为后续营销、优惠、会员等能力扩展提供了稳定承载点。
