## 订单与库存｜异步化、防超卖与一致性（喜宝）

### 技术栈与边界（简述）

- **一致性目标**：订单状态机清晰可追溯；库存防超卖；异步链路保证最终一致性。
- **关键组件**：MySQL（订单与库存）、Redis（预扣/热点保护）、MQ（削峰与异步化）、定时任务（超时释放与对账补偿）。
- **工程目标**：峰值可抗、故障可控、补偿可恢复。

### 1. 设计目标

下单链路是电商系统的“生命线”，设计目标通常是：

- **快**：峰值下单接口仍保持可接受的响应时间；
- **稳**：订单状态清晰可追溯，异常可补偿；
- **不超卖**：库存扣减要有明确的并发控制策略；
- **可扩展**：营销、优惠、风控、通知等能力可按需扩展，而不拖垮主链路。

---

### 2. 下单链路拆分：关键路径与非关键路径

为提升接口响应速度与峰值承载能力，下单链路通常拆为：

- **关键路径（同步）**：
  - 购物车校验与价格校验（防止前端缓存价格被篡改）；
  - 库存可用性校验与预扣；
  - 订单主表/明细落库（最小必要字段）；
  - 返回订单号与待支付信息。

- **非关键路径（异步）**：
  - 发送通知（短信/邮件/站内信）；
  - 写审计日志与埋点；
  - 更新统计与报表；
  - 触发营销/优惠券核销的后置步骤；
  - 生成发货任务、对接第三方系统等。

异步化通常通过 MQ 承载，避免高峰期大量非关键逻辑挤占线程与数据库连接。

---

### 3. 库存防超卖策略（组合拳）

电商库存不是单一“扣减”动作，而是一组在并发下的约束策略组合。常见落地方式：

#### 3.1 Redis 预扣库存 + 订单超时释放（预留/占用）

- 下单时先在 Redis 做预扣（原子操作），成功才允许创建订单；
- 支付成功后再进行最终扣减（或确认扣减）；
- 未支付超时则释放预扣库存（TTL 或定时任务扫描释放）。

这种模式在大促场景下能显著减轻数据库行锁争用。
同时对热点 SKU 可以做库存分片/分段预扣，降低单 key 冲突概率（按业务复杂度选择是否引入）。

#### 3.2 数据库乐观锁（version）或条件更新

支付确认或最终扣减时，在数据库侧使用乐观锁：

- `UPDATE sku SET stock = stock - ? , version = version + 1 WHERE id = ? AND stock >= ? AND version = ?`

避免并发更新造成库存被扣成负数，失败则重试或触发补偿逻辑。

#### 3.3 订单幂等与重复提交防护

用户重复点击、网络重试、前端超时重发都会导致重复下单风险：

- 以 `requestId / idempotencyKey` 作为幂等键；
- 相同幂等键只允许生成一个有效订单；
- 对支付回调也要做幂等处理，避免重复变更订单状态与库存。

#### 3.4 取消与退款的库存回补

- 订单取消（未支付/超时关闭）释放预扣库存；
- 已支付订单退款按业务规则回补库存（即时回补或批量回补），并与仓储/发货状态联动避免重复回补；
- 对“部分退款/部分发货”场景按明细维度治理库存回补。

---

### 4. 订单状态机与一致性治理

订单系统必须用明确状态机治理复杂异常：

- **典型状态**：`待支付` → `已支付` → `待发货` → `已发货` → `已完成`；中间穿插 `已取消/已关闭/退款中/已退款` 等。
- **一致性策略**：
  - 主状态变更在 DB 事务内完成；
  - 异步消息消费必须幂等（同一订单重复消费不会造成重复扣减/重复发货）；
  - 用“对账/补偿任务”处理消息丢失或消费失败的边缘情况。
  - 对关键状态变更记录操作人/来源（用户操作/系统任务/支付回调），便于审计与追责。

---

### 5. 消息队列与补偿机制

异步化带来最终一致性问题，必须配套补偿：

- **消息可靠性**：
  - 生产端确保消息落盘/确认；
  - 消费端使用手动 ack，失败重试与死信队列（DLQ）。
- **消费幂等**：
  - 以订单号/业务唯一键作为去重依据；
  - 处理“至少一次投递”导致的重复消费。
- **补偿对账**：
  - 定时扫描“状态不一致订单”（例如已支付但库存未确认）；
  - 对账后触发补偿动作（补扣库存、重发通知、重试发货任务）。

---

### 6. 风控与反作弊（下单侧约束）

- 对同一用户/同一地址/同一支付方式的异常下单频率做限制；
- 对异常设备与异常 IP 行为触发验证码/限流；
- 对“薅券/刷单”特征做规则拦截或进入人工审核队列（视业务阶段而定）。

---

### 7. 性能治理与热点保护

大促场景下热点 SKU、热点用户会形成集中冲击：

- **限流与降级**：对热点接口做限流，对非关键查询降级；
- **拆单与分批**：库存扣减与营销计算分批执行；
- **缓存保护**：防止缓存击穿（热点 key 互斥重建、空值缓存）；
- **数据库保护**：合理索引、减少跨表 join、拆分大字段与非必要写入。

---

### 8. 可观测与上线保障

- **关键指标**：下单接口耗时、库存预扣成功率、预扣释放延迟、MQ 堆积、订单状态停留时间分布、支付回调延迟、失败重试次数。
- **灰度策略**：核心改造通过开关与灰度逐步放量，出现异常可快速回退到旧链路或降级路径。
- **对账任务**：大促期间提高对账频率，及时修正异常订单与库存，避免问题累积。

---

### 9. 结果与收益

- 下单接口响应时间下降，峰值承载更稳定；
- 通过预扣 + 乐观锁 + 幂等治理，显著降低超卖与重复下单风险；
- 主链路更清晰，非关键能力可通过 MQ 持续扩展。
